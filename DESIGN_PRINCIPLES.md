# Margin Experience Design Rubric

This rubric is the default standard for future product and marketing changes. It translates the supplied Nielsen Norman Group course outlines into reviewable decisions for Margin.

## 1. Begin with the user’s job

Every page and workflow must have one primary user goal. Organize the interface around the reader’s mental model: books contain chapters; chapter checks reveal understanding; gaps lead to challenges; reviews test whether learning lasted; practice develops transferable skills.

Before adding a feature, state:

- The user problem it addresses
- The primary action or decision
- What evidence will show that it worked

## 2. Minimize visual search

- Give each screen one visually dominant next action.
- Use hierarchy, spacing, and position before adding containers.
- Keep persistent navigation stable.
- Group controls by task, not by data type or implementation.
- Use progressive disclosure for advanced, infrequent, or explanatory content.
- Do not hide essential status, consequences, or the next required step.

## 3. Make the system legible

At every meaningful interaction, show:

- Current state
- What changed
- What happens next
- Whether the action can be reversed

Use plain, actionable language. Prefer labels that describe outcomes (“Save draft,” “Schedule review”) over generic verbs (“Continue,” “Submit”) when context is not unmistakable.

## 4. Reduce cognitive and interaction cost

- Ask only for information needed at the current step.
- Preserve user input across navigation and recoverable errors.
- Prefer recognition over recall for navigation and settings.
- Keep comprehension questions effortful only when that effort serves learning.
- Avoid making users learn the interface and the reading skill simultaneously.
- Support novices with contextual guidance while keeping expert paths efficient.

## 5. Prevent errors and protect trust

- Constrain invalid choices before submission.
- Explain destructive, consequential, or AI-mediated actions before they occur.
- Provide specific recovery guidance near the problem.
- Distinguish source evidence, system inference, and user-authored interpretation.
- Never present an AI judgment as an objective grade.

## 6. Structure information for findability

- Use consistent categories, labels, and metadata across navigation, dashboards, books, and reviews.
- Validate major taxonomy changes with card sorting or tree testing.
- Keep labels concrete and user-centered.
- Provide multiple routes only when they match distinct information-seeking behaviors.
- Preserve orientation with active navigation states, headings, and visible hierarchy.

## 7. Design pages for scanning

- Match visual priority to content priority.
- Use descriptive headings and concise lead text.
- Avoid false bottoms and repetitive walls of visually identical cards.
- Ensure calls to action provide clear information scent.
- Use carousels, tabs, and accordions only when their interaction cost is justified.
- Design and verify meaningful responsive behavior rather than merely stacking elements.

## 8. Treat novelty as a hypothesis

AI interactions, conversational patterns, animation, and emerging visual styles must solve an observed problem. For each novel pattern:

- Name the benefit over a familiar convention.
- Identify likely comprehension or accessibility risks.
- Prototype the smallest credible version.
- Validate it before making it foundational.

## 9. Design for human limits

- Keep working-memory demands low.
- Direct attention without competing signals.
- Chunk long processes and show progress.
- Allow enough time to read, decide, and respond.
- Never rely on color alone to communicate state.
- Respect reduced-motion preferences and keyboard/assistive-technology use.

## 10. Definition of done

A meaningful interface change is not complete until it has been checked for:

- Clear primary task and action
- Predictable navigation and terminology
- Visible system status and useful feedback
- Error prevention and recovery
- Keyboard focus and semantic accessibility
- Responsive behavior
- Reduced-motion behavior where applicable
- Content scannability
- Consistency with the learning model
- A validation plan for unproven assumptions
